Search Trends Service

Сервис реального времени для отображения топа популярных поисковых запросов на маркетплейсе (виджет "Сейчас ищут"). Он принимает поток событий из брокера сообщений, агрегирует данные за последние 5 минут и предоставляет API для получения актуального топа с возможностью исключения нежелательных слов через стоп-лист.

ИНСТРУКЦИЯ ПО ЛОКАЛЬНОМУ ЗАПУСКУ

Требования: Docker, Docker Compose.

Запуск:
  docker-compose up --build

После запуска будут доступны:
  - Сервис: http://localhost:8080
  - NATS: nats://localhost:4222 (внутренний обмен)
  - Тестовый продюсер, отправляющий синтетические поисковые запросы каждые 10 мс

Примеры запросов к API:

Получить топ-5 запросов:
  curl "http://localhost:8080/api/v1/top?limit=5"

Добавить слово в стоп-лист:
  curl -X POST http://localhost:8080/api/v1/stoplist -H "Content-Type: application/json" -d '{"word":"bad"}'

Удалить слово из стоп-листа:
  curl -X DELETE "http://localhost:8080/api/v1/stoplist?word=bad"

Метрики Prometheus доступны по адресу:
  http://localhost:8080/metrics

КОНТРАКТ ДАННЫХ

События поступают в NATS-топик search.queries. Ожидаемый формат JSON:
{"query": "красные кроссовки", "timestamp": "2026-05-25T12:34:56Z", "user_id": "user123"}

Поля:
- query (строка, обязательно) — текст поискового запроса.
- timestamp (строка, обязательно) — время совершения запроса в формате RFC3339. Необходимо для точного попадания в 5-минутное скользящее окно, даже если событие пришло с задержкой. События старше 5 минут игнорируются.
- user_id (строка, опционально) — идентификатор пользователя. Зарезервировано для будущей реализации защиты от накруток (дедупликация запросов на уровне пользователя в пределах окна).

ОБОСНОВАНИЕ АРХИТЕКТУРЫ И СТРУКТУР ДАННЫХ

Основная задача — выдерживать высокую нагрузку на чтение (в 10–50 раз больше, чем запись) при постоянно меняющихся данных за последние 5 минут. Решение полностью in-memory, без внешней базы данных.

Скользящее окно:
- Используется кольцевой массив из 300 элементов (по количеству секунд в 5 минутах). Каждый элемент — bucket, содержащий map[string]uint64 и мьютекс для безопасной конкурентной записи.
- При поступлении события вычисляется временной слот (timestamp.Unix() % 300) и инкрементируется счётчик в соответствующем bucket'е.
- Фоновая горутина раз в секунду выполняет два действия:
  1. Очищает bucket, который только что вышел за пределы окна (текущее время минус 300 секунд).
  2. Собирает все счётчики из оставшихся 300 bucket'ов, суммирует, удаляет слова из стоп-листа и с помощью min-heap находит top-N запросов. Результат сохраняется в atomic.Value.
- Такой подход позволяет полностью разделить запись и чтение. Метод API Top() просто читает atomic.Value, не беря никаких блокировок. Это даёт очень низкую и стабильную задержку при большом количестве параллельных запросов.

Стоп-лист:
- Реализован как map[string]struct{} с sync.RWMutex.
- Управляется динамически через HTTP API. Изменения вступают в силу при следующем пересчёте топа (не позднее чем через 1 секунду). Это осознанный компромисс ради сохранения высокой производительности чтения.

Почему не используется внешнее хранилище:
- Сервис по условию стартует без данных, и все данные живут только в рамках текущего 5-минутного окна. Использование Redis или подобных систем добавило бы сетевые задержки и сложность развёртывания, не дав выигрыша при такой нагрузке. Текущая реализация на одном ядре обрабатывает миллионы событий в секунду, а пересчёт окна с сотней тысяч уникальных запросов занимает менее 15 миллисекунд.

TRADE-OFFS И БИЗНЕС-ЛОГИКА

Выявленные проблемы и принятые компромиссы:

1. Аномальные всплески (накрутки). Аналитики просили исключать их из виджета. Полноценная защита требует дедупликации по паре (user_id, query) в пределах окна, что существенно увеличило бы потребление памяти (необходимо хранить множества пользователей для каждого запроса) и усложнило бы код. На текущем этапе в контракт добавлено поле user_id, но сама дедупликация не реализована. При росте требований можно внедрить вероятностные структуры (Count-Min Sketch, HyperLogLog) или rate-limiting на уровне брокера.

2. Задержка отображения стоп-листа. Изменения, внесённые через API, применяются не мгновенно, а в момент следующего пересчёта (до 1 секунды). Для задачи администрирования это приемлемо. Если потребуется мгновенное скрытие, можно добавить фильтрацию непосредственно при чтении из atomic.Value ценой увеличения задержки ответа API.

3. Точное временное окно. Используется timestamp события, а не время получения сообщения брокером. Это гарантирует корректное попадание в окно даже при задержках доставки. События старше 5 минут отбрасываются сразу при вставке, не попадая в bucket'ы. Очистка старых слотов привязана к текущему системному времени, что синхронизирует скользящее окно с реальным временем.

4. Периодический пересчёт. Топ обновляется раз в секунду, а не непрерывно. Пользовательский виджет может отображать данные с отставанием до 1 секунды. Это плата за возможность обслуживать десятки тысяч запросов чтения в секунду без деградации.

5. Потребление памяти. Количество уникальных запросов за 5 минут на крупном маркетплейсе может достигать сотен тысяч. Каждый bucket хранит map, что в пике даёт несколько мегабайт оперативной памяти. При росте нагрузки можно рассмотреть шардирование окон или использование более компактных структур (например, trie или префиксное дерево для схожих запросов), но пока выбранный подход оптимален по соотношению скорость/память.

ПРОИЗВОДИТЕЛЬНОСТЬ И БЕНЧМАРКИ

Проведено нагрузочное тестирование с помощью встроенных бенчмарков Go и утилиты vegeta.

Вставка событий:
  BenchmarkWindowAdd-8    20000000    85 ns/op
  Около 11.7 миллионов операций в секунду на одном ядре.

Пересчёт топа (при 100 000 уникальных запросов):
  Полный цикл slide + recalcTop занимает примерно 15 мс, что укладывается в секундный интервал с большим запасом.

API чтения топа:
  При атаке vegeta с частотой 50 000 запросов в секунду в течение 30 секунд задержка P99 меньше 1 мс. Это достигается за счёт чтения из atomic.Value без блокировок.

МОНИТОРИНГ

Сервис экспортирует Prometheus-метрики на /metrics. Основные метрики:
- search_trends_events_total — общее количество обработанных валидных событий.
- search_trends_invalid_messages_total — количество некорректных сообщений (невалидный JSON, неправильный timestamp).
Также доступны стандартные метрики Go (горутины, память, сборщик мусора) и HTTP-обработчиков.

ТЕСТИРОВАНИЕ

Ключевая бизнес-логика покрыта unit-тестами. Проверяются:
- Корректность агрегации и получения топа.
- Вытеснение старых данных при скольжении окна.
- Фильтрация стоп-листа.
Запуск тестов: go test ./internal/...

ЛОКАЛЬНАЯ РАЗРАБОТКА

Для быстрого старта используется docker-compose, поднимающий NATS, основной сервис и генератор тестовых данных. Все зависимости включены в образы. Дополнительных действий не требуется.
